# 媒体协议契约：每呼叫一个 UDP socket 的裸 L16 双向流

## 功能说明

呼叫建立后，模块在 `0.0.0.0` 上绑定一个动态 UDP 端口，与 dialstring 指定
的对端 `<ip>:<port>` 做无连接双向裸 PCM 交换：

- **出方向**（主叫 → 对端）：主叫音频帧以原样小端 i16 PCM `sendto` 对端。
- **入方向**（对端 → 主叫）：socket 上收到的 PCM 喂给 `read_frame`，由
  FreeSWITCH 播放给主叫；数据不足时补静音。

无握手、无帧头、无序号、无重传——对丢包/乱序/重复完全交给 UDP 语义。

## 帧格式

| Field | Type | Length | Constraints |
|-------|------|--------|-------------|
| payload | i16 LE 采样数组 | 320 B（默认） | 160 采样 = 20 ms @ 8 kHz mono；奇数尾字节被忽略 |
| 数据报 | UDP | ≤ 2048 B | 一个数据报按一帧处理 |

默认编解码参数 L16 / 8 kHz / 20 ms / 单声道；对端必须按同一参数收发。

## 对端行为规约

对端应用必须：

1. 在 `<ip>:<port>` 监听 UDP；
2. 以 20 ms 节奏向模块端口发送 320 B 裸 L16（模块端口从来包源地址学习）；
3. **源地址必须等于协商的 `<ip>:<port>`**——任何其他源地址的包被直接
   丢弃（源过滤，防音频注入）；
4. 接受模块发来的同格式流。

最小可运行对端见模块自带的端到端校验脚本（`examples/` 下，含静音帧
校验、标记帧回环保序校验、源过滤校验，可直接作为协议参考实现）。

## 传输语义

| 主题 | 语义 |
|------|------|
| 丢包/乱序 | 不检测不补偿；`read_frame` 无数据时输出静音 |
| 接收积压 | 模块内暂存上限 ≈ 5.1 s，超出丢最旧 |
| 发送背压 | 媒体线程 → 发送任务为有界队列（256 帧）；满则丢帧并打节流警告，**不阻塞 50 Hz 媒体线程** |
| 对端不可达 | ICMP/ECONNREFUSED 等错误只记录节流日志（第 1 次与每 50 次），任务不死，对端恢复后媒体自愈 |
| 呼叫内单方向故障 | 只降级（静音/丢弃），不拆呼叫 |

## 时序说明

模块 `read_frame` 是非阻塞的：在真实 bridge 中，媒体节奏由对端腿（如
SIP/RTP）的 20 ms 定时器决定；若对端腿是 `echo` 这类自驱动应用，整条
媒体环会全速空转（数万帧/秒），属预期行为，不代表模块异常。

## 安全注意事项

- 协议无加密、无完整性校验；只在内网/受信链路使用。
- 唯一的注入防护是源地址过滤；不要依赖它对抗有路由能力的攻击者
  （UDP 源地址可伪造）。
- 模块端口是每呼叫动态分配，防火墙需放行出站 UDP 及回程包。
